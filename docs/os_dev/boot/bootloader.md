# 引导程序

> **一切的起源**
> **相关网站：[`OSDev Wiki`](http://os.dev.wiki/)**

## 什么是引导程序

**引导程序（Bootloader）是在操作系统内核运行之前执行的一段小程序。它是计算机上电后最早被执行的用户程序，负责初始化硬件设备、建立内存空间映射，将系统软硬件环境带到合适的状态，最终调用操作系统内核**<br>

**从字面拆解，`Boot`负责初始化硬件设备（存储控制器、时钟、电源管理等），`Loader`负责将程序从非易失性存储设备加载到内存中运行。**<br>

**为什么需要引导程序？ 电脑刚通电时，CPU不知道自己在哪里、内存中充满随机噪声、硬盘里的操作系统代码无法被直接读取。引导程序就是那位“前台管家”——它被固化在CPU一启动就能找到的地方，负责初始化关键硬件、清理内存空间，最后把操作系统从慢速存储器搬运到高速内存中**<br>

## 计算机启动流程

**现代计算机有两套启动流程：**<br>

1. **`Legacy`方式（`BIOS` + `MBR`）：**<br>

    * **`BIOS`（基本输入输出系统） ：固化在主板`ROM`中的固件，负责上电自检（`POST`）、硬件初始化、按启动顺序查找引导设备**
    * **`BIOS`检验存储设备前`512`字节，若以 `0x55 0xAA` 结尾则认定为`MBR`**
    * **`MBR`（主引导记录） ：磁盘第一个扇区（512字节），包含引导代码（1-446字节）、分区表（447-510字节）和结束标志（511-512字节的`0x55AA`）**
    * **`BIOS`将MBR加载到内存地址 `0x7C00` 并将控制权交给它**

2. **`UEFI`方式（`UEFI` + `GPT`）：**<br>

    * **`UEFI`（统一可扩展固件接口） ：现代固件标准，支持`GPT`分区（突破`MBR`的2TB限制）、图形界面、安全启动（`Secure Boot`）等特性**
    * **`UEFI`直接读取`GPT`分区表中的`EFI`系统分区（`ESP`） ，加载其中的 `.efi` 引导文件（如 `/EFI/BOOT/bootx64.efi`）**

???+ tip "两种方式的共同点"
    **最终都要将控制权交给引导程序，由引导程序负责加载操作系统内核**

## 引导程序的核心任务

**引导程序的终极目标只有一件事：把内核加载到内存，然后跳转过去执行。具体来说：**<br>

1. **将内核（及其所需的辅助文件）加载到内存中**

2. **为内核提供正确运行所需的信息（如内存布局、视频模式等）**

3. **切换到内核期望的运行环境（如保护模式）**

4. **将控制权转交给内核**

**其中，硬件初始化是`BootLoader`技术含量最高的环节——需要精确配置内存控制器的数百个寄存器，配置`CPU时钟频率`（通过`锁相环PLL`），初始化外存接口（`eMMC`、`NAND Flash`、`SD卡`等）。任何细微的时序偏差都可能导致系统后续运行中出现难以追踪的崩溃。**<br>

## 分阶段设计

**由于初始环境极其受限（x86 BIOS下只有512字节的MBR空间），现代BootLoader普遍采用分阶段设计：**<br>

**`Stage 1`（第一阶段）**<br>

* **用汇编语言编写（C语言运行环境尚未就绪）**

* **完成最原始的硬件初始化，设置堆栈指针**

* **为加载`Stage 2`准备`RAM`空间**

* **将`Stage 2`代码拷贝到`RAM`中并跳转执行**

**Stage 2（第二阶段）**<br>

* **通常用C语言编写**

* **实现更复杂的功能：初始化更多硬件、读取文件系统、加载内核**

* **代码具有更好的可读性和可移植性**

**以`GRUB`为例，其启动分为三个阶段：**<br>

**`Stage 1`：`MBR`中的引导代码加载`GRUB`核心（`core.img`）**<br>

**`Stage 1.5（可选）`：存储在`MBR`和分区之间的未分配空间，负责识别文件系统**<br>

**`Stage 2`：加载配置文件（`grub.cfg`），显示启动菜单，加载并执行选中的内核**<br>

## 常见引导程序

| 名称 | 采用系统(适配范围) | 特点/说明 |
| --- | --- | --- |
| **GRUB2** | **Linux/多系统** | **功能强大，支持多系统引导，是Linux最常用的引导器** |
| **LILO** | **Linux** | **早期引导器，已较少使用** |
| **systemd-boot** | **Linux** | **简洁，适合UEFI环境** |
| **U-Boot** | **嵌入式** | **嵌入式系统最流行的引导程序** |
| **Windows Boot Manager** | **Windows** | **微软/巨硬(开个玩笑)** |

## 写给开发者的实践指南

> **从哪里开始**<br>

* **[`OSDev Wiki`](http://os.dev.wiki/) 是操作系统开发者的圣经(你当然也能在这里找到)，其中的`Rolling Your Own Bootloader`页面详细描述了编写引导程序的步骤**

* **先理解背景理论再动手，不建议直接抄代码**

> **关键技术决策——编写引导程序时需要决定：**<br>

* **内核加载到哪里？ 实模式下通常加载到1MB以下（如物理地址 `0x10000`）；若内核较大，需要激活`A20 Gate`并切换到`Unreal Mode`以访问1MB以上内存**

* **内核在磁盘的什么位置？ 是`FAT分区`中的普通文件？还是连续扇区？还是直接用磁盘镜像工具写入？**

* **要给内核传递什么信息？ 内存布局、视频模式等——这些用`BIOS`调用获取比较容易，最好在实模式下完成**

> **开发和调试工具**<br>

* **`QEMU`：开源硬件虚拟化工具，可以在不烧录真实硬件的情况下测试引导程序**

* **`GDB`：配合QEMU进行源码级调试，可以单步跟踪引导程序的每条指令**

* **`NASM`：x86汇编编译器，常用于编写`Stage 1`**

???+ tip "小建议"
    **在`Windows`下配置开发环境可能稍麻烦，你当然可以使用`MinGW`或者`Cygwin`进行开发，但是安装一款`Linux`系统（显然，它应该叫发行版）进行开发无疑更加方便，`Debian`、`Ubuntu`都是不错的选择**

> **学习路径建议**
* **先理解启动流程：`BIOS/UEFI → Bootloader → Kernel` 的完整链路**

* **从简单的`Hello World`开始：用x86汇编写一个512字节的引导程序，在屏幕上打印文字**

* **实现两阶段引导：`Stage 1`用汇编，`Stage 2`用C语言**

* **加载真正的内核：将内核镜像从磁盘加载到内存并跳转执行**

* **考虑`UEFI`支持：现代系统越来越多地采用`UEFI`，学习`UEFI`应用程序开发**

## 提醒

* **不要以为引导程序很大：`x86 BIOS`模式下，第一阶段的引导代码只有512字节（含分区表和结束标志），实际代码空间不到446字节**

* **不要忽视实模式限制：`BIOS`引导时`CPU`处于16位实模式，只能访问1MB内存**

* **调试要趁早：分页初始化代码在没有异常处理机制的情况下调试会非常痛苦**

* **不是所有场景都需要自己写：如果只是想做操作系统内核，直接用`GRUB`等现成引导程序是合理的选择**
