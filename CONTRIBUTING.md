# ChineseDevDocs 贡献指南

> **声明**：**本指南由 AI `DeepSeek` 生成，`DevApotheosis`保留最终解释权。**

感谢你对 [ChineseDevDocs](https://github.com/DevApotheosis/ChineseDevDocs) 项目的关注！这份指南旨在帮助你了解如何为这个文档项目做出贡献，并确保整个贡献流程顺畅、标准化。

## 1. 项目概览
- **项目性质**：这是一个使用 **MkDocs** 构建的文档站点，所有文档内容均位于 `docs/` 目录下。
- **关键文件**：
    - `mkdocs.yml`：项目的配置文件，用于定义文档结构、主题和插件。
    - `docs/`：所有文档源文件（`.md` 格式）存放于此，请按此目录结构组织你的贡献。
    - `.github/workflows/`：包含用于自动部署的 GitHub Actions 工作流，提交前请确保不影响其正常运行。

## 2. 贡献流程
请遵循以下标准化流程提交贡献：

1.  **Fork 仓库**：在 GitHub 上将本仓库 Fork 到你的个人账户下。
2.  **克隆仓库**：将 Fork 后的仓库克隆到本地。
    ````bash
    git clone https://github.com/你的用户名/ChineseDevDocs.git
    cd ChineseDevDocs
    ````
3.  **创建分支**：为你的修改创建一个新的、描述性的分支。
    ````bash
    git checkout -b feature/你的功能描述   # 例如：feature/add-python-guide
    # 或
    git checkout -b fix/你修复的问题描述  # 例如：fix/broken-link-in-readme
    ````
4.  **本地预览（推荐）**：在提交前，建议在本地构建并预览你的更改，以确保格式正确。
    - 确保已安装 MkDocs：`pip install mkdocs`
    - 在项目根目录运行：`mkdocs serve`
    - 在浏览器中打开 `http://127.0.0.1:8000` 预览。
5.  **提交更改**：使用清晰、规范的提交信息。
    ````bash
    git add .
    git commit -m "类型(范围): 简短描述

    更详细的说明（可选）"
    ````
    - **提交信息类型**：`feat`（新功能/新文档）、`fix`（修复）、`docs`（文档本身修改）、`style`（格式调整）、`refactor`（重构）、`chore`（构建/工具变动）。范围（scope）是可选的，用于指明影响的模块，例如 `docs(guide)`。
6.  **推送并创建 Pull Request (PR)**：
    ````bash
    git push origin feature/你的功能描述
    ````
    - 然后，在 GitHub 上从你的分支向 `DevApotheosis/ChineseDevDocs` 的 `main` 分支提交 PR。
    - 在 PR 描述中，请清晰说明你的更改内容、目的以及任何相关的背景信息。

## 3. 文档规范
为确保文档的一致性和高质量，请遵循以下规范：

### 内容标准
- **语言**：使用清晰、准确、流畅的现代汉语（简体中文）。术语翻译请保持与业界通用译法一致。
- **风格**：采用客观、指导性的语气。确保文档结构清晰，善用标题、列表和代码块。
- **原创性**：贡献内容应为原创或已获授权。如引用外部资料，请注明出处。

### 格式要求
- **文件格式**：所有文档必须使用 Markdown (`.md`) 格式。
- **目录结构**：新增文档时，请将其放置在 `docs/` 目录下合适的子目录中，并更新 `mkdocs.yml` 文件中的 `nav` 部分以反映新的文档结构。
- **内部链接**：使用相对路径链接到项目内的其他文档页面（例如 `[指南](./guide.md)`）。
- **图片与资源**：如需添加图片，请将其放入 `docs/assets/` 目录下（如不存在则创建），并使用相对路径引用。

### 代码块规范
- 对于命令行、代码示例等，必须使用带有语言标识的围栏式代码块（例如 ```bash、```python），以便 MkDocs 正确进行语法高亮。**请注意**：本指南中的示例因转义需要使用了四个反引号，实际提交时请改为三个。

## 4. PR 审核流程
1.  **审核**：项目维护者将审查你的 PR，检查其内容质量、格式规范和对项目的适用性。
2.  **反馈与修改**：审核过程中可能会提出修改意见。请积极沟通，并根据反馈更新你的 PR（只需将新更改推送到你的贡献分支，PR 会自动更新）。
3.  **合并**：PR 通过审核后，将由维护者合并到主分支。合并后，自动化工作流将重新构建并部署文档站点。

## 5. 行为准则
我们期望所有贡献者都能营造一个开放、友好的协作环境。请保持尊重、专业和建设性的沟通。

## 6. 获取帮助
如果你在贡献过程中遇到任何问题，欢迎在 PR 中提出，或通过项目的 Issue 功能发起讨论。
